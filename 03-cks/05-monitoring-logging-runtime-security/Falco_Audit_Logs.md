# Falco and Audit Logs

These are the two main runtime security topics in CKS.
Falco detects malicious behavior at runtime. Audit logs track API server activity.

---

## Falco

Falco is a runtime security tool. It watches system calls and other kernel events
and fires alerts when rules match suspicious behavior.

### How Falco works

```
Container process  →  syscalls  →  eBPF/kernel module  →  Falco engine  →  Rule match  →  Alert
```

Falco is already installed in the CKS exam environment. You do not install it.
You work with its rules, config, and output.

### Key file locations

```
/etc/falco/falco.yaml               # main config: output formats, log level, rules path
/etc/falco/falco_rules.yaml         # default rules
/etc/falco/falco_rules.local.yaml   # your custom/override rules (add here)
```

### Start/restart Falco

```bash
# Falco often runs as a systemd service
systemctl status falco
systemctl restart falco
journalctl -fu falco   # live logs

# Or as a pod (check if it's in kube-system)
kubectl get pods -n kube-system | grep falco
kubectl logs -n kube-system <falco-pod> -f
```

### Read Falco output (alerts)

```bash
journalctl -fu falco | grep -i warning
# Or from log file if configured:
tail -f /var/log/falco.log
```

### Understand a Falco rule

```yaml
- rule: Terminal shell in container
  desc: A shell was used as the entrypoint/exec point into a container
  condition: >
    spawned_process
    and container
    and shell_procs
    and proc.tty != 0
    and container_entrypoint
  output: >
    A shell was spawned in a container with an attached terminal
    (user=%user.name %container.info shell=%proc.name parent=%proc.pname
    cmdline=%proc.cmdline terminal=%proc.tty container_id=%container.id image=%container.image.repository)
  priority: NOTICE
  tags: [container, shell, mitre_execution]
```

**Rule anatomy:**
| Field | Purpose |
|-------|---------|
| `rule` | Name |
| `desc` | Description |
| `condition` | What triggers it (Falco filter expression) |
| `output` | What is logged when triggered |
| `priority` | Severity: EMERGENCY, ALERT, CRITICAL, ERROR, WARNING, NOTICE, INFO, DEBUG |
| `tags` | Categories |

### Write a custom Falco rule

Always write custom rules in `/etc/falco/falco_rules.local.yaml`:

```yaml
# Detect package manager execution inside a container
- rule: Package manager launched in container
  desc: Detect apt, yum, apk running in a container
  condition: spawned_process and container and proc.name in (apt, apt-get, yum, apk, pip, pip3)
  output: "Package manager launched (user=%user.name container=%container.name image=%container.image.repository cmd=%proc.cmdline)"
  priority: WARNING
  tags: [container, process]
```

Restart Falco after editing rules:
```bash
systemctl restart falco
```

### Override an existing rule

```yaml
# In falco_rules.local.yaml — disable a built-in rule
- rule: Write below etc
  enabled: false

# Override with custom condition
- rule: Write below etc
  condition: <your modified condition>
  append: false    # replaces the original
```

### Test Falco is working

```bash
# Trigger a known rule violation in a running container:
kubectl exec -it <pod> -- bash
# This triggers "Terminal shell in container"

# Check logs for the alert
journalctl -fu falco | grep -i "shell"
```

## Kubernetes Audit Logs

Audit logs record every API request — who did what, when, on what resource.
They are generated by `kube-apiserver`.

### Audit Policy

The audit policy defines **what** to log and **how much** detail.

```yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
# Log all pod changes at RequestResponse level
- level: RequestResponse
  resources:
  - group: ""
    resources: ["pods"]

# Log secrets at Metadata level only (no content)
- level: Metadata
  resources:
  - group: ""
    resources: ["secrets"]

# Log everything from system:admin user at Request level
- level: Request
  users: ["system:admin"]

# Don't log read-only requests to nodes
- level: None
  verbs: ["get", "list", "watch"]
  resources:
  - group: ""
    resources: ["nodes"]

# Log everything else at Metadata
- level: Metadata
```

### Audit levels

| Level | What is logged |
|-------|---------------|
| `None` | Nothing logged |
| `Metadata` | Request metadata only (user, verb, resource, timestamp) |
| `Request` | Metadata + request body |
| `RequestResponse` | Metadata + request + response bodies |

### Enable Audit Logging on kube-apiserver

Edit `/etc/kubernetes/manifests/kube-apiserver.yaml`:

```yaml
spec:
  containers:
  - command:
    - kube-apiserver
    # ... other flags ...
    - --audit-policy-file=/etc/kubernetes/audit-policy.yaml
    - --audit-log-path=/var/log/kubernetes/audit.log
    - --audit-log-maxage=30          # days
    - --audit-log-maxbackup=10       # files
    - --audit-log-maxsize=100        # MB
    volumeMounts:
    - mountPath: /etc/kubernetes/audit-policy.yaml
      name: audit-policy
      readOnly: true
    - mountPath: /var/log/kubernetes/audit.log
      name: audit-log
  volumes:
  - name: audit-policy
    hostPath:
      path: /etc/kubernetes/audit-policy.yaml
      type: File
  - name: audit-log
    hostPath:
      path: /var/log/kubernetes/audit.log
      type: FileOrCreate
```

The kubelet restarts the apiserver pod automatically when you save the manifest.

### Read audit logs

```bash
# Raw log (JSON format)
cat /var/log/kubernetes/audit.log | jq .

# Filter for a specific user
grep '"user":{"username":"jane"}' /var/log/kubernetes/audit.log

# Filter for secrets access
grep '"resource":"secrets"' /var/log/kubernetes/audit.log | jq '{user:.user.username,verb:.verb,resource:.objectRef.resource,ns:.objectRef.namespace}'

# Filter by verb
grep '"verb":"delete"' /var/log/kubernetes/audit.log | jq .
```

### Audit policy stages

| Stage | When |
|-------|------|
| `RequestReceived` | When the request is received |
| `ResponseStarted` | When response headers are sent (streaming only) |
| `ResponseComplete` | When response body is complete |
| `Panic` | When a panic occurs |

```yaml
# Log at a specific stage only:
- level: Metadata
  stages:
  - ResponseComplete
  resources:
  - group: ""
    resources: ["pods"]
```

## CKS Exam Patterns

**Falco scenario:**
> "An alert shows unknown process inside a container. Identify which container and write a Falco rule to detect it."
1. Read Falco logs: `journalctl -fu falco`
2. Write rule in `/etc/falco/falco_rules.local.yaml`
3. `systemctl restart falco`
4. Verify rule triggers

**Audit log scenario:**
> "Enable audit logging to capture all secret reads. Store logs at /var/log/kubernetes/audit.log."
1. Create audit policy YAML with `level: Metadata` for secrets
2. Edit `/etc/kubernetes/manifests/kube-apiserver.yaml` to add flags and volume mounts
3. Wait for apiserver to restart
4. Verify: `cat /var/log/kubernetes/audit.log | jq . | head -20`
